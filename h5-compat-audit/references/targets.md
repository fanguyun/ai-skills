# 兼容性目标矩阵

除非用户明确给出更窄的设备范围，否则默认按本矩阵作为兼容性基线。

## 设备基线

- 安卓手机：安卓 6 及以上，重点警惕老厂商 WebView、低内存设备和弱 GPU。
- iPhone：iPhone 7 之前机型，按更老的 Safari/WebKit 能力下限评估，同时考虑更小屏幕和更紧张的内存。
- 微信浏览器：
  - iOS：主要按 WKWebView 行为评估，但叠加微信 JS-SDK 与导航怪异行为。
  - Android：同时考虑系统 WebView 与 X5 内核差异、版本碎片化，以及更明显的媒体和输入边界问题。
- 桌面端：仅考虑当前主流 Chrome、Safari、Edge；忽略 IE。

## 实际假设

### Android 6+

默认对以下内容保持风险意识：

- 未经转译或缺少 polyfill 的 ES2015+ 支持不完整
- `fetch`、`FormData`、文件上传行为存在碎片化
- flexbox 细节和老 viewport bug
- 大动画、图表、长列表性能弱
- 自动播放、内联视频和权限流程边界问题

### iPhone 7 之前机型

默认对以下内容保持风险意识：

- 屏幕更小、布局更容易拥挤
- 旧 Safari/WebKit 对 CSS 特性的支持缺口
- 图片、canvas、长页面可用内存更紧张
- `position: fixed`、软键盘遮挡、滚动回弹问题
- 视频自动播放和内联播放限制

即使在新 iPhone 上表现正常，也要为布局、滚动和媒体行为准备更保守的降级方案。

### 微信内浏览器

必须始终考虑：

- JS-SDK 初始化时机和签名依赖
- 分享、OAuth、支付、跳 App 等流程
- 被拦截或被改写的导航行为
- 上传下载限制和预览行为
- 字体缩放、viewport、history 的怪异行为

不要假设微信和系统 Safari 或 Chrome 的行为一致。

## 兼容性判定原则

以下情况可以接受：

- 视觉效果有节制地降级
- 动画质量降低
- sticky、磨砂、玻璃等效果退化为更简单样式
- 高级 API 回退为基础交互

以下情况不能接受：

- 白屏
- hydration 失败或 JS 解析失败
- 表单提交或上传被阻断
- 无法滚动、无法点击或无法触达核心操作
- 微信里的登录、分享或支付流程失效
- 关键操作被布局遮挡
